# 发布前本地修复计划

## 目标

把当前软件从“可用原型”修到“可以放心交给别人使用”的状态。

本计划不包含“支持更多远程控制软件”。ToDesk、AnyDesk、TeamViewer 等检测覆盖放到发布后迭代。

## 不做的事

- 不重写现有业务架构。
- 不改向日葵当前检测规则，除非发现它阻塞本计划。
- 不新增复杂账号系统。
- 不把公网访问改成手工命令行流程。

## 修复顺序

### 1. 修复乱码和文案

问题：

- 部分 `.razor`、`.cs` 文件里的中文已经变成乱码，界面会出现 `????` 或奇怪文字。
- 部分提示语太长，像写给开发者看的，不适合普通操作者。

要做：

- 全项目搜索乱码字符和异常中文。
- 修复 Dashboard、Sidebar、Settings、公开状态页、TunnelService 中的乱码。
- 统一提示语：短、清楚、只说用户需要做什么。

验收：

- 界面不再出现 `????`、乱码中文、错误符号。
- 关键提示不超过两行。
- 错误提示能直接说明下一步操作。

### 2. 修复公网状态可信度

问题：

- 现在 cloudflared 进程启动后，软件就可能显示“公网已连接”。
- 但公网地址可能仍然打不开，例如 Cloudflare 1033、502、503。

要做：

- 隧道启动后自动请求 `https://域名/status`。
- 只有公网请求成功返回状态页，才显示“公网已连接”。
- 请求失败时显示“公网不可用”，并展示简短原因。
- 读取 cloudflared 的标准输出和错误输出，不能只读一边。
- `config.yml` 中 Windows 路径要正确转义或加引号。

验收：

- 公网打不开时，软件不能显示“已连接”。
- 公网能打开时，设置页和左下角显示“公网已连接”。
- cloudflared 报错时，软件能显示简短错误原因。

### 3. 迁移数据目录

问题：

- 当前数据库、配置、日志可能写在程序安装目录。
- 如果安装到 `Program Files`，普通用户可能没有写权限。

要做：

- 将可变数据目录统一迁到用户可写位置。
- 建议位置：`%ProgramData%\Monitor`。
- 需要迁移这些文件：
  - `monitor.db`
  - `config.json`
  - `logs`
  - `backups`
  - cloudflared 相关本地配置如需要也要明确位置
- 首次启动时，如果旧位置有数据，要迁移到新位置。

验收：

- 软件安装到受保护目录后仍能正常写入数据库和配置。
- 旧数据不会丢失。
- 设置页能显示当前数据目录。

### 4. 加强登记码安全

问题：

- 操作人登记码现在要求太短。
- 公开登记接口没有失败次数限制。
- 登记码明文保存在配置中。

要做：

- 登记码最少 8 个字符。
- 登记码只保存哈希，不保存明文。
- 公网登记接口加入失败次数限制。
- 连续失败后短时间锁定。
- 设置页只显示“已设置 / 未设置”，不显示明文。

验收：

- 少于 8 位不能保存。
- 配置文件中看不到登记码明文。
- 错误登记码连续尝试会被锁定。
- 正确登记码仍能登记操作人。

### 5. 加强数据库可靠性

问题：

- 数据库缺少索引。
- 远程会话没有数据库级别的“只能有一个活跃会话”约束。
- 批量删除历史记录使用拼接 SQL。

要做：

- 给常用查询增加索引：
  - `usage_records(start_time)`
  - `usage_records(user_name)`
  - `reservations(date, status)`
  - `remote_sessions(is_active, start_time)`
- 给远程会话增加约束，避免多个活跃会话同时存在。
- 批量删除改为参数化或事务逐条删除。
- 增加简单的数据库版本记录，后续迁移可追踪。

验收：

- 历史记录很多时，列表和统计仍能正常打开。
- 数据库层面不允许出现多个活跃远程会话。
- 批量删除不再拼接 SQL。

### 6. 修复定时器重入风险

问题：

- 远程检测用周期 Timer。
- 如果一次检测卡住，下一次检测可能重叠执行。

要做：

- 远程检测改成单线程循环。
- 可用 `PeriodicTimer` 或执行锁。
- 同一时间只允许一次检测运行。
- 检测超时要暴露为异常状态，不要静默跳过。

验收：

- 即使检测耗时超过间隔，也不会并发写远程会话。
- 日志中不会出现同一时间重复创建远程会话。

### 7. 收敛公开状态页

问题：

- 公开页展示了检测信号、检测信息等技术字段。
- 这不是普通访问者需要看的内容。

要做：

- 公开页只展示：
  - 当前是否可连接
  - 远程工具
  - 操作人
  - 开始时间
  - 已占用时长
  - 最近远程记录
  - 操作人登记入口
- 移除检测信号、置信度、来源、最后检测、内部错误细节。
- 公开页视觉风格与软件主界面保持一致。

验收：

- 普通用户打开页面，3 秒内能判断能不能连接。
- 页面不暴露内部检测信号。
- 手机宽度下不溢出。

### 8. 精简设置页提醒

问题：

- 设置页提醒较多，容易打断用户。
- 有些提醒重复表达同一个问题。

要做：

- 只保留会阻塞核心功能的提醒。
- 同类问题合并成一条。
- 每条提醒只写“问题 + 去哪里处理”。

验收：

- 设置页顶部最多显示 3 条提醒。
- 没有重复提醒。
- 每条提醒都能直接指导用户完成下一步。

### 9. 补齐关键操作反馈

问题：

- 公网配置、授权、保存设置等操作需要更明确的反馈。

要做：

- 按钮执行中显示“处理中”。
- 成功后显示简短成功状态。
- 失败后显示简短原因。
- 不使用长段说明。

验收：

- 用户点击按钮后知道软件正在做什么。
- 操作失败时知道下一步要做什么。

### 10. 构建和回归检查

必须检查：

- `dotnet build` 通过。
- 打开软件后首页无乱码。
- 向日葵占用中能自动记录远程会话。
- 操作人可登记。
- 断开后远程历史保留。
- 公网页 `/status` 能打开。
- 公网不可用时软件不显示“已连接”。
- 设置页能保存管理员密码、Web 密码、登记码和公网域名。
- 备份能创建。
- 历史记录能删除。
- 预约冲突仍能阻止。

完成标准：

- 构建 0 错误。
- 核心页面无明显错位。
- 核心功能路径全部通过。
- 没有新引入的乱码。

## 建议执行方式

一次执行时按顺序做，不要跳着修：

1. 先修乱码。
2. 再修公网状态可信度。
3. 再迁移数据目录。
4. 再修安全和数据库。
5. 最后做 UI 提醒收敛和回归检查。

每完成一组修复，都要先构建，再继续下一组。
